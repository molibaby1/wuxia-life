# P0 核心体验止血 PRD

## 1. 背景

项目当前已具备基础工程可运行性，类型检查、测试和构建均可执行。但从实际游玩视角看，P0 级问题仍会直接破坏体验或削弱系统正确性：

- 主游戏界面残留生产可见的调试信息。
- 多结果选择分支存在条件判定不可信的问题。
- 事件触发存在连续复读现象，人生模拟节奏不自然。
- 游戏状态存在多套入口，后续存档、UI、测试和引擎行为可能分裂。

本 PRD 只定义 P0 阶段的止血方案，不进入 P1/P2/P3 的玩法扩展和大规模重构。

## 2. 目标

1. 让生产游戏界面不再暴露调试信息。
2. 让玩家选择的多结果分支按真实条件判定。
3. 让事件系统具备基础重复抑制和可观测验收指标。
4. 明确唯一游戏状态源，并完成旧状态链路的可控迁移或退场。
5. 为每个 P0 修复建立可重复验证方式。

## 3. 非目标

- 不新增剧情线、门派、身份或结局内容。
- 不重做整体 UI 视觉风格。
- 不重构整个事件系统为新架构。
- 不清理全部历史文档。
- 不处理 P1 的完整内容质量门禁。
- 不处理 P2 的路线深度和选择反馈重做。

## 4. 范围

### 4.1 In Scope

- 主游戏界面调试信息移除或迁移。
- DebugPanel 与生产 UI 的职责边界确认。
- 多结果分支表达式条件判定修复。
- 多结果分支测试补齐。
- 事件重复问题复现、定义、抑制和指标化。
- 状态源审计、决策、迁移、旧链路退场。
- P0 验证命令与验收报告补充。

### 4.2 Out of Scope

- 大规模视觉改版。
- 新增事件内容规模扩张。
- 全量事件格式迁移。
- 条件表达式安全执行器的完整替换。
- 存档版本迁移体系的完整设计。
- 复杂 AI 自动游玩策略重写。

## 5. 用户故事

### US-001：移除主界面生产调试块

作为玩家，我希望游戏主界面只展示游戏内容，而不是开发调试字段，这样我可以沉浸在武侠人生流程中。

验收标准：
- 主游戏界面不再展示 `currentNode`、`choices`、`lastOutcomeText` 等调试文本块。
- 选择、继续、事件结果展示功能不丢失。
- 构建通过。

优先级：1

### US-002：保留开发者诊断入口

作为开发者，我希望必要的诊断信息仍可在 DebugPanel 或明确调试入口查看，这样移除生产调试 UI 不会降低排障能力。

验收标准：
- 调试信息只在明确调试入口中显示。
- 默认游戏流程不显示诊断面板。
- 调试入口开关行为不变或更清晰。

优先级：2

### US-003：补充主流程 UI 验收记录

作为维护者，我希望 P0 UI 清理有可复现验收方式，这样后续不会重新把调试文本漏到生产界面。

验收标准：
- 记录手动或自动验收步骤。
- 验收步骤覆盖新游戏、选择事件、继续事件、结束流程中的至少两个关键节点。
- 文档或测试说明不包含本地绝对路径。

优先级：3

### US-004：统一多结果分支条件判定入口

作为玩家，我希望选择结果根据真实条件触发，而不是默认命中第一个表达式分支，这样选择才有意义。

验收标准：
- 表达式条件不再被默认视为通过。
- 前端选择结果判定复用核心条件评估能力或等价受控入口。
- 无条件分支仍可作为兜底分支。
- 条件不可解析时不会静默命中错误结果。

优先级：4

### US-005：补齐多结果分支测试

作为开发者，我希望多结果选择分支有测试覆盖，这样条件判定回归能被及时发现。

验收标准：
- 覆盖条件成功、条件失败、兜底分支、顺序优先级。
- 至少包含一个表达式条件场景。
- 默认测试门禁通过。

优先级：5

### US-006：验证自动判定事件选项条件

作为开发者，我希望 autoResolve 事件的选项过滤和结果判定一致，这样自动事件不会绕开选择条件。

验收标准：
- autoResolve 只会选择可用选项。
- autoResolve 的结果分支同样走真实条件判定。
- 存在失败时可读的错误或警告，不静默吞掉问题。

优先级：6

### US-007：建立事件复读复现样本

作为设计者，我希望能稳定复现连续受伤、生病、商机等复读现象，这样修复不是凭感觉调权重。

验收标准：
- 形成固定随机种子或固定模拟输入。
- 复现报告能列出连续重复事件或同类事件。
- 复现方式可在本地命令中运行。

优先级：7

### US-008：定义重复事件判定标准

作为设计者，我希望明确什么算重复，这样事件抑制不会误伤正常剧情推进。

验收标准：
- 区分同一事件重复、同类事件重复、同剧情线重复。
- 明确允许重复和禁止连续重复的边界。
- 标准覆盖伤病、经济、婚恋、身份、主线、日常事件。

优先级：8

### US-009：实现基础重复抑制策略

作为玩家，我希望短期内不要连续遭遇同一类无意义事件，这样人生流程更像故事而不是复读。

验收标准：
- 同一事件遵守 `maxTriggers` 和 `cooldown`。
- 同类高负面事件具备短期抑制或权重衰减。
- 日常事件与剧情事件的重复策略清晰区分。
- 不破坏必须触发的主线或关键剧情。

优先级：9

### US-010：补充事件重复率验收

作为维护者，我希望事件复读问题有指标化验收，这样后续事件扩展不会重新污染节奏。

验收标准：
- 模拟报告输出重复事件统计。
- 至少统计相邻重复、短窗口同类重复、重复最多事件。
- P0 验收阈值写入测试或文档。

优先级：10

### US-011：审计当前状态源入口

作为维护者，我希望先知道项目里有哪些状态源入口和调用点，这样统一状态源不会误删有效流程。

验收标准：
- 列出旧 store、新 composable、全局 gameEngine、存档、调试面板、演示组件的状态关系。
- 标记每个入口是主流程、演示流程、废弃流程还是待确认流程。
- 不修改业务代码。

优先级：11

### US-012：确定唯一主状态源决策

作为开发者，我希望 P0 阶段明确唯一主状态源，这样后续修复都围绕同一个状态模型展开。

验收标准：
- 明确唯一主状态源。
- 明确允许保留的只读适配层。
- 明确不再新增的旧入口。
- 决策记录写入 P0 交付说明。

优先级：12

### US-013：迁移主流程对旧状态的依赖

作为玩家，我希望开始、游玩、结局、重开流程都使用一致状态，这样 UI 和引擎不会显示不同人生。

验收标准：
- 主流程不再依赖旧状态源。
- 结局界面读取状态与游戏引擎一致。
- 重开游戏能清理主状态和事件状态。
- 类型检查通过。

优先级：13

### US-014：隔离或删除废弃状态链路

作为维护者，我希望废弃状态链路被明确隔离或删除，这样后续不会误用旧路径。

验收标准：
- 废弃入口没有主流程引用。
- 若暂不删除，必须标记为演示或历史兼容用途。
- 无未使用导入或明显死链路。
- 测试和构建通过。

优先级：14

### US-015：补充状态一致性回归测试

作为开发者，我希望状态源统一后有回归测试，这样存档、结局、重开流程不会悄悄分裂。

验收标准：
- 覆盖新游戏、推进事件、选择事件、结局或重开中的关键状态一致性。
- 默认测试门禁通过。
- 测试失败时能定位到状态同步问题。

优先级：15

### US-016：P0 收口验证与交付说明

作为项目负责人，我希望 P0 阶段完成后有清晰证据，这样可以决定是否进入 P1。

验收标准：
- `npm run typecheck` 通过。
- `npm run build` 通过。
- `npm test` 通过。
- P0 交付说明包含改动清单、验证结果、残余风险、后续建议。

优先级：16

## 6. 实施顺序

推荐按以下顺序执行：

1. US-001 至 US-003：先处理低风险 UI 调试暴露。
2. US-004 至 US-006：修复选择分支正确性。
3. US-007 至 US-010：复现并抑制事件复读。
4. US-011 至 US-015：统一状态源，控制风险逐步迁移。
5. US-016：整体收口验证。

说明：状态源统一风险最高，应在前面三个问题具备验证保护后再实施。

## 7. 验证计划

P0 阶段默认验证命令：

```bash
npm run typecheck
npm run build
npm test
```

如果某个故事新增专用脚本或模拟命令，需要在该故事交付说明中补充命令、输入和判定标准。

## 8. 风险与缓解

### 风险 1：调试信息移除后排障困难

缓解：保留 DebugPanel 或明确开发者诊断入口，不把诊断能力散落在生产 UI。

### 风险 2：多结果条件修复改变现有剧情结果

缓解：先补测试和样例，再修判定。对条件失败场景要求显式可见，避免静默命中。

### 风险 3：重复抑制误伤主线或关键剧情

缓解：先定义重复标准，主线和关键事件单独白名单或优先级保护。禁止只靠随意调低权重解决。

### 风险 4：状态源统一影响结局、存档或演示组件

缓解：先审计再迁移，主流程优先，演示组件可后置标记。每步迁移后运行类型检查和主测试门禁。

### 风险 5：P0 范围膨胀

缓解：不新增内容、不重做 UI、不全量重构事件系统。发现 P1/P2 问题只记录，不纳入本阶段实施。

## 9. 回滚策略

- UI 调试清理：回滚相关组件改动即可恢复。
- 条件判定修复：保留测试样例，若回滚必须同时记录失败原因，不允许恢复默认通过逻辑。
- 重复抑制：以小范围策略开关或局部函数回滚为单位，避免改动事件数据大面积回滚。
- 状态源统一：按故事分批提交，主流程迁移和废弃链路删除分开处理，必要时只回滚最后一批迁移。

## 10. 审批边界

本文档和配套 Ralph JSON 仅定义 P0 阶段方案。未获得明确审批前，不进入业务代码修改。

每个 user story 完成后都应补充验证证据；若跨会话继续，必须先读取本文档和配套 JSON。
