# 游戏项目改进优先级清单

## 背景

本文档用于承接项目点评后的后续改造工作，作为 AI 编码任务的总索引。

当前项目可以完成类型检查、测试与构建，但核心体验和工程结构仍存在明显问题：生产界面残留调试信息，事件触发有复读现象，多套状态链路并存，部分选择分支条件并未真实判定。后续改造应避免一次性大范围重写，而是按优先级逐项生成修改方案、审批后实施。

## 优先级原则

- P0：先修影响真实游玩和正确性的硬伤。
- P1：改善核心事件体验和内容生产质量。
- P2：增强玩法结构，让每局人生形成清晰路线。
- P3：整理表现层与工程结构，降低长期维护成本。

## P0 必须先修

### 1. 移除生产界面的调试信息

问题：主游戏界面直接展示 `currentNode`、`choices`、`lastOutcomeText` 等调试文本，破坏游戏沉浸感。

价值：让项目从开发 Demo 状态向可游玩产品状态靠拢。

后续方案要求：
- 明确移除或封装哪些调试 UI。
- 确认调试能力是否迁移到已有 DebugPanel。
- 给出界面验收方式。

### 2. 统一游戏状态源，废掉旧状态链路

问题：项目中存在旧 store、新 composable、全局 gameEngine 等多套状态来源，容易导致 UI、逻辑、测试和存档状态不一致。

价值：降低后续所有玩法改造的风险。

后续方案要求：
- 梳理当前所有状态入口和实际使用路径。
- 明确唯一主状态源。
- 标记可删除、可保留、需迁移的旧链路。
- 不在未审批前直接删除旧代码。

### 3. 修复多结果分支条件判定

问题：多结果分支中的表达式条件当前可能被默认视为通过，导致玩家选择结果不可信。

价值：让选择真正影响结果，避免“假分支”。

后续方案要求：
- 统一前端选择结果判定与核心条件评估器。
- 补充表达式条件失败、成功、多分支顺序的测试。
- 明确条件不可解析时的处理规则。

### 4. 修复事件重复与复读问题

问题：模拟过程中出现连续触发同一类事件的现象，例如连续受伤、连续生病、连续商机。

价值：这是人生模拟核心体验问题，不修会让游戏像事件池复读机。

后续方案要求：
- 用模拟日志复现重复事件。
- 区分同一事件重复、同类事件重复、同剧情线重复。
- 明确冷却、互斥、权重衰减或内容标签策略。
- 补充重复率验收指标。

## P1 核心体验提升

### 5. 重构事件选择节奏系统

问题：事件优先级、权重、冷却、日常事件、年度压力等规则混杂，最终触发行为不够可控。

价值：让人生节奏有起伏，避免随机塞事件。

后续方案要求：
- 梳理事件选择全流程。
- 明确正式事件、日常事件、剧情线事件的选择优先级。
- 给出可观测的节奏指标。

### 6. 建立事件内容质量门禁

问题：事件内容规模扩大后，重复标题、无效条件、空效果、同年龄拥堵、剧情线断裂等问题会持续污染体验。

价值：让内容扩展进入可验证流程。

后续方案要求：
- 新增或完善事件校验脚本。
- 输出清晰的问题列表和严重级别。
- 将核心校验接入现有验证命令或单独命令。

### 7. 统一事件格式和效果格式

问题：事件数据存在多套字段和兼容格式，例如 `autoEffects`、`effects`、`content.autoEffects` 等。

价值：减少玄学 bug，让事件作者知道唯一正确写法。

后续方案要求：
- 盘点所有现存格式。
- 选择目标标准格式。
- 制定迁移顺序和校验规则。

### 8. 修复条件表达式执行方式

问题：条件表达式通过动态函数执行，与“安全解析”的设计目标不一致。

价值：提高安全性、可测试性和错误定位能力。

后续方案要求：
- 明确支持的表达式语法范围。
- 替换为可控解析或白名单求值。
- 补充恶意输入、非法表达式、正常表达式测试。

## P2 玩法结构增强

### 9. 重做玩家选择的反馈闭环

问题：选择后的反馈过于泛化，玩家难以感知代价、收益和长期影响。

价值：强化“我在塑造人生”的体验。

后续方案要求：
- 定义选择反馈展示规则。
- 区分即时数值变化、关系变化、路线变化和长期旗标。
- 避免把纯数值日志直接堆给玩家。

### 10. 强化人生路线与身份系统的互斥和延续

问题：商人、侠客、门派、魔道、官场等路线需要更清晰的推进和排斥规则。

价值：让每局人生形成可辨认的命运形状。

后续方案要求：
- 梳理现有身份、路线、阵营字段。
- 明确主路线、副路线、冲突路线规则。
- 给出路线完成率和断线率验收方式。

### 11. 补充真实游玩模拟测试

问题：现有测试能证明函数和流程大体可跑，但不足以证明游戏体验稳定。

价值：用数据衡量 0-80 岁人生是否合理。

后续方案要求：
- 统计事件重复率、选择率、死亡率、路线完成率、结局分布。
- 支持固定随机种子。
- 输出便于人工阅读的模拟摘要。

### 12. 存档系统接入真实主流程

问题：若存档状态与新引擎状态不一致，人生模拟的长期体验会不可靠。

价值：保障玩家进度和长期游玩。

后续方案要求：
- 确认当前 SaveManager 的实际接入情况。
- 验证新游戏、保存、读取、继续、重开流程。
- 补充状态版本和迁移策略。

## P3 表现和工程整理

### 13. 清理报告型文档和过期方案

问题：历史报告和方案数量较多，容易干扰后续判断当前真实状态。

价值：降低 AI 和人工维护时的上下文噪音。

后续方案要求：
- 不直接删除文档，先分类。
- 区分当前有效文档、历史归档文档、可废弃文档。
- 建立一个当前文档入口。

### 14. 重整 UI 信息架构

问题：属性、关系、路线、事件、选择反馈堆在同一屏，层次感不足。

价值：让界面从表单面板转向武侠人生体验。

后续方案要求：
- 明确主界面核心信息优先级。
- 区分常驻状态、事件叙事、选择操作、历史回顾。
- 保持现有功能不丢失。

### 15. 拆分超大引擎类

问题：核心引擎承担事件选择、状态、难度、身份、挫折、历史记录等过多职责。

价值：降低后续改动风险。

后续方案要求：
- 先画出现有职责边界。
- 选择低风险拆分点。
- 每次拆分必须有等价测试保护。

### 16. 建立内容编辑和校验脚本入口

问题：事件作者缺少统一命令来校验事件、模拟人生和输出异常。

价值：让内容扩展从手工撞运气变成稳定流水线。

后续方案要求：
- 设计统一命令入口。
- 聚合格式校验、内容质量校验、模拟测试。
- 输出可读报告和机器可读结果。

## 推荐执行批次

第一批只处理 P0：

1. 移除生产界面的调试信息。
2. 修复多结果分支条件判定。
3. 修复事件重复与复读问题。
4. 统一游戏状态源，废掉旧状态链路。

说明：状态源统一风险较高，可以先做只读梳理和方案，再审批实施。调试 UI、分支条件、事件复读更容易形成可验证闭环，适合作为第一批落地任务。

## 后续使用方式

后续每次从本文档选择一个条目，单独生成修改方案。修改方案必须包含：

- 目标与非目标
- 涉及文件
- 具体修改步骤
- 验收标准
- 验证命令
- 潜在风险
- 回滚方式

未审批修改方案前，不进入业务代码实施。
