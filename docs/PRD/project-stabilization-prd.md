# 武侠人生项目工程化修复 PRD

## 1. 背景
当前项目处于“功能迭代快于工程治理”的失衡状态，核心表现为：
- 主分支构建失败，无法稳定交付
- 测试入口不标准，无法形成 CI 防线
- 类型系统存在系统性漂移，模块接口口径不一致
- 仓库生成物污染严重，协作噪音高

该状态会导致任何新功能迭代都呈现“增量返工”，工程成本非线性上升。

## 2. 目标
在不新增玩法功能的前提下，将项目恢复到可持续迭代状态。

### 2.1 业务目标
- 支持稳定发布与回归验证
- 降低多人协作冲突与回归概率

### 2.2 工程目标
- 构建通过率达到 100%
- TypeScript 错误清零
- 测试具备标准入口与可执行基线
- 仓库生成物隔离，工作区噪音显著下降
- 游戏引擎入口收敛为单一主路径

## 3. 范围

### 3.1 In Scope
- 工程治理与质量基线修复
- 类型系统与状态模型统一
- 引擎入口收敛（双入口去重）
- 最小测试基线建立
- CI 最小闭环落地

### 3.2 Out of Scope
- 新剧情、新系统、新数值玩法
- 美术与交互重设计
- 大规模性能专项优化（仅保留必要稳定性修复）

## 4. 用户故事（按优先级）

### US-1（P0）作为开发者，我需要仓库可干净协作
- 我希望执行测试/构建后不会产生大量未跟踪噪音文件
- 我希望生成物和源码分离，便于 code review 与合并

### US-2（P0）作为开发者，我需要一键验证项目健康度
- 我希望通过标准命令执行类型检查、构建、测试
- 我希望本地结果与 CI 结果一致

### US-3（P0）作为开发者，我需要类型系统可信
- 我希望核心模块类型口径一致，避免“编译期盲区”
- 我希望新增逻辑可以被类型系统及时拦截风险

### US-4（P1）作为开发者，我需要单一路径驱动游戏引擎
- 我希望前端只连接一个引擎入口，避免行为漂移
- 我希望事件推进逻辑可追踪、可中断、可验证

### US-5（P1）作为团队，我需要最小回归保护
- 我希望关键流程（开局、事件推进、选择、结局）有稳定 smoke 测试
- 我希望 PR 在入主干前被自动质量门禁拦截

## 5. 详细方案（可并行且低耦合）

### WP-A：仓库卫生与输出隔离（低耦合）
- 目标：解决生成物污染与协作噪音
- 主要动作：
  - 补齐 `.gitignore` 的报告产物覆盖规则（包含 `public/reports/**`）
  - 明确测试/模拟输出目录规范（统一到 `artifacts/` 或既有专用目录）
  - 清理已被跟踪的临时产物
- 输出：仓库卫生规范与规则落地

### WP-B：脚本与命令入口标准化（低耦合）
- 目标：建立统一执行入口
- 主要动作：
  - 在 `package.json` 增加并规范 `test`、`typecheck`、`build`、`test:all`
  - 将现有脚本纳入标准命令链路并统一退出码
  - 约束脚本输出格式，便于 CI 解析
- 输出：本地/CI 一致的命令入口

### WP-C：TypeScript 错误清零（高优先级，可按目录并行）
- 目标：恢复编译可信性
- 主要动作：
  - 分片修复 `src/components/**`、`src/composables/**`、`src/core/**`、`src/data/**`
  - 统一 effect 类型枚举与字段口径（如大小写与命名一致性）
  - 消除隐式 any、空值不安全访问、错误模块路径
- 输出：`typecheck` 与 `build` 全绿

### WP-D：状态模型统一（中耦合）
- 目标：消除 `GameState/PlayerState/EventRecord` 口径分裂
- 主要动作：
  - 明确单一真源类型定义与导出位置
  - 统一 `flags/events/connections` 访问方式
  - 收敛条件评估器的输入契约，移除不可靠推断路径
- 输出：核心状态模型一致

### WP-E：引擎入口收敛（中耦合）
- 目标：避免双引擎并行造成行为漂移
- 主要动作：
  - 明确保留一个引擎入口（旧入口下线或适配过渡）
  - 前端调用统一到单一路径
  - 事件推进改为可控流程，避免递归式失控
- 输出：唯一引擎主路径

### WP-F：最小测试基线（中低耦合）
- 目标：建立可回归、可阻断的基础测试保护
- 主要动作：
  - 建立 smoke 测试集（开局、推进、选择、结局）
  - 增补关键回归测试（5-10 条）
  - 接入标准测试命令
- 输出：稳定可执行的最小回归网

### WP-G：CI 最小闭环（低耦合）
- 目标：防止带病代码进入主分支
- 主要动作：
  - CI 串联 `typecheck + build + test`
  - PR 质量门禁启用（失败即阻断）
- 输出：自动化质量闸门

## 6. 验收标准（可测试）

### 6.1 P0 验收
- `npm run typecheck` 返回 0 且无 TS 错误
- `npm run build` 返回 0
- `npm test` 返回 0（至少 smoke 套件通过）
- 执行测试后 `git status` 不再出现大规模报告噪音

### 6.2 P1 验收
- 前端仅通过单一引擎入口运行主流程
- 核心状态类型只存在一个真源定义
- CI 对 PR 自动执行质量门禁并阻断失败构建

### 6.3 质量指标
- TS 错误：359 -> 0
- 关键流程 smoke 覆盖：0 -> 100%（四条主链路）
- 构建通过率：<100% -> 100%

## 7. 里程碑与排期

### M1：工程基线恢复（Day 1-2）
- 完成 WP-A、WP-B
- 启动 WP-C（components/composables 分片）

### M2：编译全绿（Day 3-4）
- 完成 WP-C（core/data 分片）
- 完成 WP-D

### M3：运行链路收敛（Day 4-5）
- 完成 WP-E
- 完成 WP-F

### M4：交付闭环（Day 5-6）
- 完成 WP-G
- 验收与发布建议输出

## 8. 资源与工时评估

### 8.1 总量评估
- 单人总工时：9-14 人日
- 3 人并行日历时间：4-7 天
- 风险缓冲：20%

### 8.2 分包工时
- WP-A：0.5-1 人日
- WP-B：0.5-1 人日
- WP-C：4-6 人日
- WP-D：1.5-2.5 人日
- WP-E：1-2 人日
- WP-F：1-2 人日
- WP-G：0.5-1 人日

## 9. 风险与缓解
- 风险 1：历史事件数据格式不一致导致连锁改动
  - 缓解：先建立 schema 映射与校验脚本，再逐步收敛
- 风险 2：双引擎切换产生行为回归
  - 缓解：先落 smoke，再迁移入口
- 风险 3：治理期间插入新需求导致返工
  - 缓解：冻结功能迭代窗口，治理优先

## 10. 回滚策略
- 分工作包小步提交，任何包可独立回滚
- 引擎入口收敛采用“可回退开关”短期兜底（仅迁移窗口期）
- CI 门禁分阶段启用：先告警后阻断（不超过一个迭代）

## 11. 依赖关系与并行矩阵
- 可完全并行：WP-A、WP-B、WP-G（G 在收尾接入）
- 高并行主战场：WP-C 可按目录分片并行
- 轻依赖：WP-D 依赖 WP-C 的核心类型收敛结果
- 轻依赖：WP-E 参考 WP-D 的状态模型口径
- 可穿插：WP-F 可从 Day 2 起先落 smoke，再补全

## 12. 发布与收口
- 发布前必须满足：typecheck/build/test 全绿
- 交付材料：
  - 改动清单
  - 验收记录（命令与结果）
  - 残余风险与后续建议（若有）
