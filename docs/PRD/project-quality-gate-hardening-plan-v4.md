# 武侠人生项目质量门禁加固方案（V4）

## 1. 背景与问题证据
最新验证显示存在“假绿门禁”风险：
- 真实门禁执行后退出码为 0
- 但同一日志内出现多次表达式求值错误（Failed to evaluate expression / ReferenceError）
- 仍存在测试环境警告（localstorage-file path invalid）

结论：当前质量门禁仅基于进程退出码，缺乏关键错误文本拦截，导致“有错放行”。

## 2. 目标
不改玩法功能，仅加固验证体系，将项目从“结果绿”升级为“过程可置信的绿”。

## 3. 范围

### 3.1 In Scope
- 门禁规则加固方案设计
- 错误/告警分级与阻断标准
- 稳定性验证方案（多次模拟）
- CI 收口策略与执行清单

### 3.2 Out of Scope
- 业务剧情新增
- 数值玩法重构
- 大规模架构重写

## 4. 修改方案（执行顺序）

### P0-1：门禁规则从“退出码”升级为“退出码 + 关键日志”
- 保留现有真实测试集（AllTests / IntegrationTests / testGameSimulation）
- 新增关键错误词阻断策略：
  - Failed to evaluate expression
  - ReferenceError
  - Unknown stat
  - localstorage-file invalid path
- 命中任一关键词即判定失败

验收：
- 出现关键错误词时门禁返回失败
- 无关键错误词时维持正常通过

### P0-2：错误/告警分级规范
- 设立三级规则：
  - Blocker：必须失败（运行时异常、表达式求值失败、未知核心属性）
  - Warning：允许通过但必须记录（非关键性能提示）
  - Info：仅展示
- 形成统一关键词字典，避免团队口径漂移

验收：
- 测试输出可按级别分类
- Blocker 与 Warning 边界清晰

### P0-3：测试环境噪音清零方案
- 明确 localstorage 参数初始化规范（本地/CI）
- 规范环境变量与默认路径
- 防止环境配置警告掩盖真实故障

验收：
- 测试执行日志不再出现 localstorage 参数路径警告

### P1-1：稳定性门禁方案（批量模拟）
- 在 testGameSimulation 基础上定义批量运行规范（建议 20 次）
- 设置“关键错误词零容忍 + 指标边界”双门槛
- 任一轮触发 Blocker 即失败

验收：
- 批量运行结果可重复复现
- 失败场景有明确定位信息

### P1-2：CI 与发布流程收口
- CI 必跑：typecheck + build + real test gate + stability gate
- 发布前检查项加入“关键错误词为 0”
- 更新团队文档与发布清单

验收：
- PR 质量门禁可自动阻断
- 发布前检查具备统一标准

## 5. 风险与缓解
- 风险 1：初期阻断率上升影响迭代速度
  - 缓解：分阶段启用（先告警后阻断，窗口不超过 1 个迭代）
- 风险 2：关键词策略误判
  - 缓解：关键词白名单/黑名单可配置并评审
- 风险 3：稳定性批量测试增加耗时
  - 缓解：分层执行（快速门禁 + 夜间完整门禁）

## 6. 工作量评估

- P0-3：0.5 人日
- P1-1：0.5-1 人日

## 7. 交付标准
1. 质量门禁策略文档化并可执行
2. 关键错误词阻断规则明确
3. 稳定性门禁方案可落地
4. CI 与发布检查项完成收口
