# 游戏开发流程规范

**版本**: v1.0  
**生效日期**: 2026-03-12  
**状态**: 强制执行

---

## 一、总则

### 1.1 目的
为确保游戏开发质量，规范开发流程，防止新开发内容引入缺陷，特制定本规范。

### 1.2 适用范围
本规范适用于游戏开发的所有阶段，包括但不限于：
- 功能模块开发
- 系统集成
- 性能优化
- Bug 修复
- 内容扩展

### 1.3 基本原则
1. **测试先行**: 每个阶段完成后必须立即测试
2. **质量门禁**: 测试不通过不得进入下一阶段
3. **问题优先**: 发现问题立即停止开发，优先修复
4. **文档记录**: 所有测试结果必须记录存档

## 二、开发阶段划分

### 2.1 标准开发流程

```
┌─────────────┐
│  需求分析   │
└──────┬──────┘
       ↓
┌─────────────┐
│  方案设计   │
└──────┬──────┘
       ↓
┌─────────────┐
│  编码实现   │
└──────┬──────┘
       ↓
┌─────────────┐
│  单元测试   │ ← 必须
└──────┬──────┘
       ↓
┌─────────────┐
│  集成测试   │ ← 必须
└──────┬──────┘
       ↓
┌─────────────┐
│ 整体测试    │ ← 强制（新增）
└──────┬──────┘
       ↓
┌─────────────┐
│  测试评审   │ ← 强制
└──────┬──────┘
       ↓
┌─────────────┐
│ 进入下一阶段│
└─────────────┘
```

### 2.2 阶段定义

#### 阶段 1: 功能模块开发
- **输入**: 需求文档、设计方案
- **活动**: 编码实现、单元测试
- **输出**: 功能模块、单元测试报告
- **质量门禁**: 单元测试覆盖率≥80%

#### 阶段 2: 系统集成
- **输入**: 功能模块、接口定义
- **活动**: 模块集成、接口测试
- **输出**: 集成系统、集成测试报告
- **质量门禁**: 所有接口测试通过

#### 阶段 3: 整体测试（新增 - 强制）
- **输入**: 集成系统
- **活动**: 运行整体测试脚本
- **输出**: 整体测试报告
- **质量门禁**: 所有测试用例通过（100% pass rate）

#### 阶段 4: 测试评审
- **输入**: 整体测试报告
- **活动**: 评审测试结果、确认质量
- **输出**: 评审结论、进入下一阶段许可
- **质量门禁**: 无严重缺陷、评审通过

## 三、整体测试要求

### 3.1 测试覆盖范围

整体测试必须覆盖以下四个方面：

#### 1. 核心功能测试
- ✅ 事件执行器功能
- ✅ 条件评估器功能
- ✅ 状态管理系统
- ✅ 时间系统
- ✅ 数据存储系统

**测试重点**:
- 功能正确性
- 边界条件处理
- 错误处理机制

#### 2. 用户交互流程测试
- ✅ 完整事件流程
- ✅ 选择分支流程
- ✅ 事件链执行
- ✅ 状态变更流程

**测试重点**:
- 流程连贯性
- 用户体验
- 数据一致性

#### 3. 性能测试
- ✅ 事件执行速度（<5ms/次）
- ✅ 条件评估速度（<2ms/次）
- ✅ 内存使用（<50MB）
- ✅ FPS（≥60）

**测试重点**:
- 响应时间
- 资源占用
- 稳定性

#### 4. 兼容性测试
- ✅ 数据格式版本兼容
- ✅ 向后兼容
- ✅ 跨平台兼容（未来）

**测试重点**:
- 版本迁移
- 数据兼容

### 3.2 测试执行频率

| 场景 | 频率 | 说明 |
|------|------|------|
| 功能模块完成后 | 每次 | 每个功能模块开发完成后立即执行 |
| 集成完成后 | 每次 | 模块集成完成后立即执行 |
| 性能优化后 | 每次 | 性能优化完成后立即执行 |
| Bug 修复后 | 每次 | Bug 修复完成后立即执行 |
| 定期测试 | 每日 | 每天下班前执行一次完整测试 |

### 3.3 测试执行方式

#### 方式 1: 命令行执行
```bash
# 运行所有测试
npx tsx tests/AllTests.ts

# 运行特定测试套件
npx tsx tests/AllTests.ts --suite=core
npx tsx tests/AllTests.ts --suite=performance
```

#### 方式 2: IDE 集成
在 IDE 中配置测试运行器，一键执行测试。

#### 方式 3: CI/CD 集成
在 CI/CD 流水线中集成测试，自动执行。

## 四、测试结果处理

### 4.1 测试报告

每次测试执行后自动生成测试报告，内容包括：

**基本信息**:
- 测试时间戳
- 测试套件数量
- 测试用例总数
- 通过/失败/跳过数量
- 通过率
- 总耗时

**性能指标**:
- 平均事件执行时间
- 平均条件评估时间
- 内存使用
- FPS

**问题列表**:
- 严重问题（critical）
- 主要问题（major）
- 次要问题（minor）

### 4.2 测试通过标准

#### 通过标准（必须全部满足）:
1. ✅ 所有测试用例通过（pass rate = 100%）
2. ✅ 无严重缺陷（critical issues = 0）
3. ✅ 性能指标达标
4. ✅ 内存使用在合理范围内

#### 不通过情况:
- ❌ 任一测试用例失败
- ❌ 存在严重缺陷
- ❌ 性能指标不达标
- ❌ 内存使用超标

### 4.3 测试不通过处理流程

```
测试不通过
    ↓
立即停止开发
    ↓
记录问题详情
    ↓
分析根本原因
    ↓
制定修复方案
    ↓
实施修复
    ↓
重新运行测试
    ↓
测试通过？──否──→ 继续修复
    ↓
   是
    ↓
进入下一阶段
```

**关键要求**:
1. **立即停止**: 测试不通过时，立即停止后续开发工作
2. **优先修复**: 将修复失败测试用例作为最高优先级任务
3. **重新测试**: 修复后必须重新运行所有测试
4. **记录存档**: 所有测试报告必须记录存档

## 五、角色与职责

### 5.1 开发人员

**职责**:
- 开发完成后立即运行测试
- 修复失败的测试用例
- 确保测试通过后再提交代码
- 记录测试结果

**要求**:
- 熟悉测试框架
- 能独立运行测试
- 能快速定位和修复问题

### 5.2 测试人员

**职责**:
- 编写和维护测试用例
- 执行整体测试
- 分析测试结果
- 跟踪问题修复
- 生成测试报告

**要求**:
- 精通测试框架
- 能设计全面的测试用例
- 能准确评估测试质量

### 5.3 项目经理

**职责**:
- 确保开发流程得到遵守
- 监督测试执行
- 评审测试结果
- 批准进入下一阶段

**要求**:
- 熟悉测试流程
- 能评估测试报告
- 能做出质量决策

## 六、测试框架使用

### 6.1 测试框架结构

```
tests/
├── GameTestFramework.ts    # 测试框架核心
├── AllTests.ts             # 所有测试用例
├── reports/                # 测试报告目录
│   └── test-report-YYYY-MM-DD.json
└── utils/                  # 测试工具
```

### 6.2 编写测试用例

```typescript
import { GameTestFramework, TestSuite, assert } from './GameTestFramework';

const myTestSuite: TestSuite = {
  testCases: [
    {
      name: '测试用例名称',
      description: '测试用例描述',
      test: async () => {
        // 测试代码
        const result = someFunction();
        assert(result === expected, '期望结果与实际不符');
      },
    },
  ],
};

const framework = new GameTestFramework();
framework.registerSuite('我的测试套件', myTestSuite);
```

### 6.3 运行测试

```bash
# 运行所有测试
npx tsx tests/AllTests.ts

# 查看测试报告
cat tests/reports/test-report-*.json
```

## 七、文档与记录

### 7.1 测试报告存档

所有测试报告必须存档，保存期限：
- 开发阶段测试报告：项目结束后 3 个月
- 发布版本测试报告：永久保存

### 7.2 测试报告格式

测试报告应包含：
- 测试基本信息
- 测试结果详情
- 性能指标
- 问题列表
- 修复建议

### 7.3 测试日志

每次测试执行应记录日志，包括：
- 测试时间
- 测试人员
- 测试环境
- 测试结果
- 备注说明

## 八、违规处理

### 8.1 违规行为

以下行为视为违规：
- ❌ 跳过测试直接进入下一阶段
- ❌ 测试不通过继续开发
- ❌ 伪造测试结果
- ❌ 不记录测试结果

### 8.2 处理措施

根据违规严重程度：
- 轻微违规：口头警告
- 一般违规：书面警告
- 严重违规：项目除名
