# P1 事件质量与节奏治理 PRD

## 1. 背景

P0 阶段聚焦核心体验止血：生产调试信息、多结果分支、事件复读、状态源一致性。用户已说明 P0 在其他会话中完成实施。P1 阶段不继续扩写玩法内容，而是治理事件系统的可控性和内容生产质量。

当前 P1 关注四类问题：

- 事件选择节奏由优先级、权重、冷却、日常事件、年度压力等规则混合决定，实际行为不够透明。
- 事件内容规模增加后，重复标题、无效条件、空效果、同年龄拥堵、剧情线断裂等问题会持续污染体验。
- 事件和效果数据存在多套字段与兼容格式，内容作者缺少唯一正确写法。
- 条件表达式执行方式需要从动态执行转向可控、可测试、可诊断的求值路径。

本 PRD 定义 P1 阶段的具体方案与 Ralph user stories。未获得审批前，不进入业务代码实施。

## 2. P1 目标

1. 建立事件选择节奏的可观测指标和分层策略。
2. 建立事件内容质量门禁，阻止低质量事件继续进入主流程。
3. 统一事件和效果字段标准，减少兼容分支和数据歧义。
4. 用受控条件求值替代不透明的动态执行路径。
5. 将 P1 质量检查接入可重复验证命令。

## 3. 非目标

- 不新增大规模剧情内容。
- 不重做 UI 信息架构。
- 不设计 P2 的人生路线深度系统。
- 不接入完整存档版本迁移体系。
- 不拆分整个核心引擎类。
- 不一次性迁移所有历史文档。

## 4. 前置条件

P1 实施前应确认：

- P0 相关 user stories 已完成或被项目负责人明确豁免。
- 默认验证命令可运行。
- 当前事件模拟或测试可以给出基本人生流程样本。

若 P0 在当前工作树中缺少可验证证据，P1 的第一项任务必须先补基线确认，不应直接改动事件节奏或数据格式。

## 5. 范围

### 5.1 In Scope

- 事件选择全流程梳理和指标定义。
- 正式事件、日常事件、剧情线事件、关键事件的优先级与节奏边界。
- 事件内容质量校验规则和严重级别。
- 事件格式与效果格式的标准化目标定义。
- 旧字段盘点、迁移策略和校验规则。
- 条件表达式语法范围、受控求值器、错误诊断和测试。
- P1 质量门禁命令或脚本入口。

### 5.2 Out of Scope

- 扩写新门派、新身份、新结局。
- 把所有事件内容一次性重写。
- 全量删除兼容字段。
- 大规模视觉重构。
- 将所有测试框架替换为新框架。

## 6. 用户故事

### US-001：确认 P0 基线与 P1 入口条件

作为维护者，我希望在进入 P1 前确认 P0 已完成并且默认验证可运行，这样 P1 不会建立在未收口的基础上。

验收标准：
- 读取 P0 PRD 和 P0 实施证据或交付说明。
- 记录 P0 入口状态：完成、部分完成、豁免或阻塞。
- 若 P0 证据不足，明确列出 P1 受影响故事。
- 不修改业务代码。

优先级：1

### US-002：梳理事件选择全流程

作为开发者，我希望理解事件从候选池到最终触发的完整路径，这样节奏调整有明确落点。

验收标准：
- 梳理候选加载、年龄过滤、条件过滤、冷却、剧情线、日常事件、权重、年度压力、触发记录的顺序。
- 标出每一步的输入、输出和失败回退路径。
- 标出正式事件与日常事件的切换条件。
- 产出流程说明或图示。

优先级：2

### US-003：定义事件节奏指标

作为设计者，我希望事件节奏可以被量化，这样“节奏好不好”不是主观争吵。

验收标准：
- 定义每年龄事件数、空窗年龄数、正式事件占比、日常事件占比、选择事件占比、剧情线连续度、关键事件延迟等指标。
- 指标包含目标区间或 P1 观察基线。
- 指标输出能关联到模拟样本。

优先级：3

### US-004：建立事件节奏基线报告

作为维护者，我希望先获得当前节奏基线，这样后续改动能对比是否变好。

验收标准：
- 使用固定随机种子或固定模拟输入生成基线报告。
- 报告覆盖至少一段完整人生样本或多个代表性年龄段。
- 报告输出 P1 指标。
- 报告可以通过本地命令重新生成。

优先级：4

### US-005：制定事件分层选择策略

作为设计者，我希望关键事件、剧情线事件、普通正式事件和日常事件各有明确选择边界，这样系统不会随机塞事件。

验收标准：
- 定义事件层级：关键事件、剧情线事件、正式普通事件、日常事件、兜底事件。
- 定义各层级进入候选和回退的条件。
- 明确哪些事件可以跳过，哪些事件必须保护。
- 不要求立即大规模改数据。

优先级：5

### US-006：实现低风险节奏选择调整

作为玩家，我希望事件触发节奏更稳定，这样人生流程有起伏而不是堆叠或空转。

验收标准：
- 选择逻辑按已批准的分层策略执行。
- 保持关键事件可触发。
- 日常事件只在正式事件不足或节奏需要时介入。
- 节奏基线报告显示指标变化。
- 默认测试门禁通过。

优先级：6

### US-007：补充节奏回归测试

作为开发者，我希望事件节奏有回归测试，这样后续调权重不会重新破坏人生节奏。

验收标准：
- 测试覆盖空候选回退、关键事件保护、日常事件介入、候选权重选择。
- 测试不依赖不可控随机结果，或使用固定随机种子。
- 默认测试门禁通过。

优先级：7

### US-008：盘点事件内容质量问题类型

作为内容维护者，我希望知道事件数据常见问题类型，这样质量门禁能覆盖真实风险。

验收标准：
- 盘点重复标题、重复 ID、无效年龄范围、空内容、空效果、无效条件、无效属性、同年龄拥堵、剧情线断裂、不可达事件。
- 每类问题给出严重级别建议。
- 不修改事件内容。

优先级：8

### US-009：定义事件质量门禁规则

作为维护者，我希望事件内容质量有明确规则，这样新增内容不会靠人工肉眼检查。

验收标准：
- 定义 blocker、major、minor 三类严重级别。
- 明确哪些问题必须阻断验证命令。
- 明确哪些问题只输出警告和报告。
- 规则写入文档或校验配置。

优先级：9

### US-010：实现事件质量校验脚本

作为开发者，我希望有命令可以扫描事件数据并输出结构化问题列表，这样内容问题可重复发现。

验收标准：
- 脚本扫描事件入口中实际加载的数据。
- 输出事件 ID、文件或来源、问题类型、严重级别、说明。
- blocker 问题导致命令失败。
- 无需启动前端即可运行。

优先级：10

### US-011：接入事件质量门禁到验证流程

作为维护者，我希望事件质量校验进入默认或明确的验证流程，这样内容质量不会游离在发布之外。

验收标准：
- 明确新增命令名称。
- 文档说明何时运行该命令。
- 若接入默认验证命令，需保证当前基线可通过或有明确豁免清单。
- 默认测试门禁仍通过。

优先级：11

### US-012：盘点事件与效果现存格式

作为开发者，我希望先知道事件和效果字段有多少种写法，这样统一格式不会误删有效数据。

验收标准：
- 盘点事件顶层字段、content 字段、autoEffects、effects、choice.effects、choice.outcomes、requirements、thresholds、triggerConditions 等现存格式。
- 统计每种格式使用数量。
- 标记主流程使用、历史兼容、疑似废弃三类。
- 不修改事件数据。

优先级：12

### US-013：定义目标事件格式标准

作为内容作者，我希望有唯一推荐事件格式，这样新增事件不用猜字段写法。

验收标准：
- 定义目标事件结构。
- 定义目标效果结构。
- 定义选择事件和自动事件的标准写法。
- 明确弃用字段和迁移窗口。
- 标准写入文档。

优先级：13

### US-014：实现格式校验与迁移提示

作为开发者，我希望旧格式能被识别并提示迁移，这样格式统一可以分批推进。

验收标准：
- 校验脚本能识别目标格式、旧格式、混合格式。
- 对旧格式输出迁移提示。
- 对危险混合格式输出 blocker 或 major。
- 不自动大规模改写事件数据。

优先级：14

### US-015：迁移首批低风险事件格式

作为内容维护者，我希望先迁移低风险事件样本，这样可以验证标准格式可行。

验收标准：
- 选择少量低风险事件作为样本。
- 迁移后行为等价。
- 相关校验通过。
- 默认测试门禁通过。

优先级：15

### US-016：定义条件表达式语法范围

作为开发者，我希望条件表达式支持范围明确，这样求值器可以安全、可测、可诊断。

验收标准：
- 明确支持的属性访问、flag 查询、event 查询、比较运算、逻辑运算、括号规则。
- 明确不支持的语法。
- 明确非法表达式的错误行为。
- 写入文档或类型说明。

优先级：16

### US-017：实现受控条件求值器

作为维护者，我希望条件表达式不依赖动态函数执行，这样条件判定更安全、更稳定。

验收标准：
- 受控求值器覆盖 P1 定义的表达式语法。
- 不使用动态函数执行表达式。
- 错误包含表达式、位置或原因等可诊断信息。
- 现有合法表达式能通过测试。

优先级：17

### US-018：替换运行时条件求值路径

作为玩家，我希望事件条件和选择条件走同一套可靠判定，这样剧情触发一致且可解释。

验收标准：
- 运行时事件条件使用受控求值器。
- 选择条件、阈值或触发条件之间的职责边界清晰。
- 条件失败不静默误触发事件。
- 默认测试门禁通过。

优先级：18

### US-019：替换测试与模拟中的动态求值

作为开发者，我希望测试和模拟工具不再使用另一套动态求值逻辑，这样验证结果和运行时一致。

验收标准：
- 测试框架和模拟工具复用运行时条件求值能力。
- 不保留独立的动态函数求值分支。
- 模拟报告仍可生成。
- 默认测试门禁通过。

优先级：19

### US-020：补充条件表达式安全与回归测试

作为维护者，我希望非法表达式和边界条件有测试覆盖，这样求值器不会悄悄退回危险实现。

验收标准：
- 覆盖合法表达式、非法字段、非法语法、flag 查询、event 查询、逻辑组合、括号优先级。
- 覆盖恶意或非白名单语法不会执行。
- 默认测试门禁通过。

优先级：20

### US-021：生成 P1 质量门禁报告

作为项目负责人，我希望 P1 结束时有质量报告，这样可以判断是否进入 P2。

验收标准：
- 报告包含节奏指标、内容质量问题数量、格式校验结果、条件表达式测试结果。
- 报告包含 P1 前后对比或基线说明。
- 报告列出残余风险和 P2/P3 候选事项。

优先级：21

### US-022：P1 收口验证

作为维护者，我希望 P1 完成后有完整验证证据，这样后续会话可以安全接力。

验收标准：
- `npm run typecheck` 通过。
- `npm run build` 通过。
- `npm test` 通过。
- P1 新增质量命令通过或输出已审批豁免。
- P1 PRD JSON 中完成的 user stories 更新验证状态或交付说明记录完成证据。

优先级：22

## 7. 推荐实施顺序

1. US-001：确认 P0 基线。
2. US-002 至 US-007：先让事件节奏可观测、可回归，再调整选择策略。
3. US-008 至 US-011：建立内容质量门禁。
4. US-012 至 US-015：统一格式标准并迁移低风险样本。
5. US-016 至 US-020：替换条件表达式执行方式。
6. US-021 至 US-022：报告与收口验证。

说明：条件求值器替换风险较高，应在内容质量与格式边界更清晰之后实施。

## 8. 验证计划

P1 默认验证命令：

```bash
npm run typecheck
npm run build
npm test
```

P1 实施中新增的质量命令应满足：

- 可在本地直接运行。
- 输出人类可读摘要。
- 对 blocker 问题返回失败状态。
- 可被后续 CI 或发布门禁复用。

## 9. 风险与缓解

### 风险 1：事件节奏调整破坏关键剧情触发

缓解：先定义关键事件保护层，再调整普通事件和日常事件。每次修改后跑固定种子模拟。

### 风险 2：质量门禁一次性暴露大量历史问题

缓解：区分 blocker、major、minor。只让真正影响运行或核心体验的问题阻断，其他问题进入报告和迁移清单。

### 风险 3：格式统一导致历史事件不可用

缓解：先盘点和定义标准，再迁移低风险样本。禁止无报告的大规模自动改写。

### 风险 4：条件求值器替换引发剧情触发差异

缓解：先锁定语法范围和测试样本，对现有合法表达式建立回归测试，再替换运行时路径。

### 风险 5：P1 范围膨胀到玩法扩写

缓解：P1 只治理质量、节奏、格式和条件执行。新增剧情、路线深度、UI 重构全部进入后续阶段。

## 10. 回滚策略

- 事件节奏调整：按选择策略函数或配置变更回滚，保留基线报告用于对比。
- 质量门禁：若阻断过严，可回滚严重级别配置，不回滚问题发现能力。
- 格式迁移：首批样本小范围迁移，按事件文件或事件 ID 回滚。
- 条件求值器：保留测试样本，若替换失败，回滚运行时接入点而不是删除求值器代码。

## 11. 审批边界

本文档和配套 Ralph JSON 仅定义 P1 阶段方案。未获得明确审批前，不进入业务代码修改。

跨会话继续 P1 时，必须先读取本文档和配套 JSON，并按 user stories 优先级顺序实施。
